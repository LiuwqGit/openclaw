# Root Cause Analysis — Issue #103956

## Symptom

Users who set `session.pruneAfter: "1d"` in openclaw.json expect sessions older than 1 day to be automatically removed. Instead, all sessions are retained indefinitely regardless of `pruneAfter`.

## Root Cause Location

- File: `src/config/sessions/store-load.ts` line 463 (original)
- Function: `loadSessionStore`

## Root Cause Description

In `loadSessionStore()`, the stale-entry pruning (`pruneStaleEntries`) is gated by `Object.keys(store).length > maintenance.maxEntries`. This means age-based pruning only runs when the store already exceeds the entry count limit. If the user has fewer sessions than `maxEntries` (default 500), pruning never fires regardless of how old sessions get — `pruneAfter` acts only as an eligibility cutoff, not a trigger.

Compare with `applyEnforcedMaintenance()` in `store-maintenance-operations.ts`, which runs `pruneStaleEntries` unconditionally (when mode is `"enforce"`) and only gates the capping step by `shouldRunSessionEntryMaintenance`.

## Trigger Condition

- `session.maintenance.mode` is `"enforce"` (the default)
- Session count is below `maxEntries` (default 500)
- The store has sessions older than `pruneAfter`

## Three Candidate Solutions

### Solution A (Recommended): Separate pruning from entry-count gate
- **Description**: Remove the `&& Object.keys(store).length > maintenance.maxEntries` condition from the pruning block. `pruneStaleEntries` runs unconditionally when mode is `"enforce"`. The capping step remains gated by `shouldRunSessionEntryMaintenance`.
- **Scope**: Only `src/config/sessions/store-load.ts` (change one if-condition, delete one redundant `preserveSessionKeys` redeclaration)
- **Proof level reachable**: L2 (vitest)
- **Rationale**: Matches the established pattern in `applyEnforcedMaintenance()`. Minimal diff (3 lines changed, 1 removed). No behavioral change to capping.
- **Risk**: Low
- **Complexity**: Low

### Solution B: Add a cron-based unconditional sweep
- **Description**: Add a separate timer or cron job that runs `pruneStaleEntries` independently of store-write time
- **Scope**: New source file + config surface
- **Rationale**: More robust but introduces new infrastructure
- **Risk**: Medium (new timer, new config)
- **Complexity**: Medium

### Solution C: Lower default maxEntries
- **Description**: Reduce `DEFAULT_SESSION_MAX_ENTRIES` to a lower number so the existing gate fires sooner
- **Scope**: `src/config/sessions/store-maintenance.ts`
- **Rationale**: Works indirectly, but doesn't solve the conceptual mismatch — `pruneAfter` would still not be an independent trigger
- **Risk**: Medium (could cause unexpected session capping for users near the new limit)
- **Complexity**: Low

## Selection Decision

**Chosen**: Solution A
**Reason**: Minimal diff, zero new infrastructure, exactly matches the pattern already proven in `applyEnforcedMaintenance()`. The capping behavior (`shouldRunSessionEntryMaintenance`) remains unchanged. The only behavior change is that `pruneStaleEntries` now runs whenever mode is `"enforce"`, regardless of entry count.
